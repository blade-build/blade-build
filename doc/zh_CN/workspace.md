# 工作空间配置

## 工作空间定义

Blade 要求项目有一个明确的根目录，作为工作空间（workspace）。该工作空间是 C++ 代码中 `#include` 路径解析的基准，所有 `#include` 指令都以工作空间根目录为参照。

## 工作空间带来的收益

### 头文件管理

- **避免命名冲突：** 通过从根开始的绝对路径，消除头文件同名问题
- **层级清晰：** 为头文件组织提供清晰的层级结构
- **依赖明确：** 让代码库中的包含路径解析无歧义

### 库管理

- **库身份唯一：** 通过工作空间路径避免库重名
- **依赖追踪精确：** 支持精准的依赖解析与版本管理
- **构建优化：** 链接期间的库解析更清晰

### 开发效率

- **便于定位文件：** 统一的路径结构让文件查找更简单
- **构建更快：** 依赖解析更高效
- **团队协作：** 跨开发环境的项目结构保持一致

## 存量项目迁移

对于已经存在自有包含路径规范的项目，Blade 提供了[迁移脚本](../../tool/fix-include-path.sh)，帮助平滑接入工作空间模型。

## 工作空间探测机制

### 动态探测

Blade 采用动态探测策略，而非依赖配置文件或环境变量，这样便于在多个并行的工作空间之间切换。

**探测算法：**

- **自底向上搜索：** 从当前目录开始向上查找 `BLADE_ROOT` 文件
- **根目录识别：** `BLADE_ROOT` 所在目录即为当前工作空间
- **任意子目录可用：** 无论当前位于多深的子目录，都能正确识别

### 创建工作空间

在根目录下放置一个 `BLADE_ROOT` 文件，即可创建工作空间：

```console
touch BLADE_ROOT
```

## 工作空间组织策略

### 单仓库（monorepo）开发

在 monorepo 模式下，将 `BLADE_ROOT` 放在仓库根目录，建立统一的工作空间：

```console
$ ls -1
BLADE_ROOT
common/
thirdparty/
xfs/
xcube/
torca/
your_project/
...
```

### 多仓库（polyrepo）开发

在多仓库模式下，每个仓库各自创建 `BLADE_ROOT`，拥有独立的工作空间：

```console
# 仓库 A
$ ls -1
BLADE_ROOT
src/
libs/

# 仓库 B
$ ls -1
BLADE_ROOT
components/
services/
```

## 最佳实践

- **结构一致：** 各工作空间保持统一的目录组织
- **绝对路径：** `#include` 指令一律使用基于工作空间的路径
- **纳入版本控制：** 将 `BLADE_ROOT` 文件纳入版本控制，确保团队一致
- **迁移规划：** 对于存量项目，借助迁移脚本平滑过渡
